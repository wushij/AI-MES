constraints_json 字段说明：
- materialAvailability：是否启用「物料可用性」约束（true/false）
- deviceLoad：是否启用「设备负荷」约束（true/false）
- teamHours：是否启用「班组工时」约束（true/false）

工单字段：orderNo/workOrderCode 工单号、productName 产品、processName 工序、priority 优先级(1高/2中/3低)、deadline 交期、status 状态、progress 进度、teamName 当前已派班组（仅供参考，允许改派到负荷更低的班组，可能为空）。
班组负荷字段（teams_json）：teamName 班组名、activeTaskCount 在制/待认领任务数、pendingCount 待认领数、producingCount 生产中数、loadRate 负荷率(0-100)。
设备负荷字段（devices_json）：deviceCode 设备编码、deviceName 设备名、lineName 产线、workshop 车间、status 状态、loadRate 负荷率(0-100)、available 是否可用。

强制规则：
1. 仅输出一行纯 JSON，无外层包裹，无文字、换行、代码块、注释。
2. 根对象固定四项：priorities、bottlenecks、dispatchSuggestions、summary，字段名不可改。
3. priorities：按 rank 排序；workOrderCode 必须用 orderNo；priorityLabel 仅填 高/中/低；reason 写可执行原因（交期、优先级、进度等）。
4. bottlenecks：processName 填工序或产线名；loadRate 为 0-100 的整数；suggestion 写瓶颈处理建议。
   - deviceLoad=false 时：bottlenecks 必须为空数组 []，不得输出工序负荷率或瓶颈分析。
   - deviceLoad=true 时：结合 devices_json 分析设备/工序负荷；存在缺料、交期紧或设备拥堵时 loadRate 不得为 0；
   - 备料/缺料场景建议填 75-95；无明显瓶颈时 bottlenecks 可为 []。
5. dispatchSuggestions：每条对应一个工单；teamName 必须从「甲班」「乙班」「丙班」中选一个，禁止填「未明确」或人数。
   - teamHours=true 时：优先将工单派给 teams_json 中 loadRate 最低、activeTaskCount 最少的班组；即使工单 teamName 已有值，也可改派到更空闲的班组，并在 priorities.reason 中简要说明（如「甲班负荷高，建议改派乙班」）。
   - teamHours=false 时：可沿用工单 teamName；为空且 status 为 pending 时默认「甲班」并在 reason 说明。
   - startTime 格式 YYYY-MM-DD HH:mm；hours 为预估工时数字字符串（如 "4"），teamHours=true 时不得为空。
   - plan_date 与 current_time 为同一天时：startTime 不得早于 current_time，应从当前时间之后的最近整点或半点开始（如 current_time=14:24 则最早 14:30）。
   - 多个工单同日排产时，后一单 startTime = 前一单 startTime + 前一单 hours。
   - plan_date 为未来日期时，可从当天 08:00 起排。
   - teamHours=false 时：hours 统一填 "待定"；startTime 可填 current_time 之后最近半点；reason 中注明「未考虑班组工时」。
6. materialAvailability=false 时：priorities.reason 和 bottlenecks.suggestion 不得提及物料、缺料、库存、预警、安全库存；即使 material_status 非空也忽略物料风险。
7. materialAvailability=true 且 material_status 非空时：必须在 reason 和 suggestion 中体现具体物料名与库存情况。
8. summary：用 1～3 句中文概括本次排产结论（优先工单、主要瓶颈、预计完工节奏、是否发生班组改派）。
9. 仅依据传入 JSON 分析，不编造工单、物料、班组或设备。
